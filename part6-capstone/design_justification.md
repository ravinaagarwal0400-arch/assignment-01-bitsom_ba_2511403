## Storage Systems

I would use a hybrid architecture because the hospital has four very different workloads. For predicting readmission risk, I would store historical treatment, encounter, diagnosis, and medication data in a data lakehouse. This gives cheap long-term storage, supports feature engineering, and works well for ML training on large historical datasets. For doctors querying patient history in plain English, I would keep the authoritative patient record in an OLTP relational database such as PostgreSQL, and build a separate retrieval layer using a document store plus a vector database. Clinical notes, discharge summaries, lab reports, and imaging summaries can be embedded and indexed for semantic retrieval, while the answer shown to the doctor is grounded in the source records. For monthly management reports, I would load curated data from operational systems into a data warehouse with dimensional models for occupancy, costs, admissions, and department performance. This is the right system for repeatable, governed reporting. For real-time ICU vitals, I would use a streaming platform such as Kafka and a time-series database optimized for high-ingest sensor data. That supports continuous writes, recent trend analysis, and alerting without overloading the transactional clinical database.

## OLTP vs OLAP Boundary

The transactional system ends at the operational clinical layer: patient registration, encounters, medication orders, diagnoses, lab events, and other day-to-day hospital workflows belong in the OLTP relational database. Those systems prioritize ACID consistency, low-latency writes, and correctness. The analytical system begins once data is replicated out through ETL, ELT, or CDC pipelines into the lakehouse and warehouse. From that point onward, the data is used for historical analysis, BI dashboards, model training, and derived predictions rather than direct care transaction processing. The vector index also sits outside core OLTP because it is a derived access layer, not the source of truth.

## Trade-offs

A major trade-off is architectural complexity. This design uses multiple specialized systems: relational OLTP, lakehouse, warehouse, vector search, streaming, and time-series storage. That increases operational overhead, data governance burden, and synchronization challenges. I would mitigate this by making the relational EHR data the clear system of record, using CDC-based pipelines for freshness, enforcing strong metadata/catalog controls, and defining strict data contracts between teams. I would also limit duplication to high-value use cases only, so each storage system exists for a clear reason rather than becoming unnecessary sprawl.
